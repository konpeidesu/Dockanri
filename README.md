# Document Lifecycle Manager

> GitHubのドキュメント版

Document Lifecycle Manager（DLM）は、社内で利用する提案資料、製品紹介資料、POV資料、導入手順書、操作手順書などを、常に最新の状態に保つためのドキュメント管理アプリです。

現在Excelで管理している資料台帳をアプリ化し、更新状況の可視化、担当者管理、レビュー・承認フロー、更新履歴の記録を実現します。

GitHubのIssue、Pull Request、Commit Historyの考え方を参考にしながら、非開発者でも利用しやすいUIを提供します。

## 背景

現在のExcelによるドキュメント管理には、以下の課題があります。

### 更新状況が分からない

- 更新が必要か
- 誰が対応しているか
- 対応が完了しているか

これらを一覧で把握することが困難です。

### 更新履歴が残らない

- いつ更新されたか
- 誰が更新したか
- 何を変更したか

といった履歴を追跡できません。

### 責任者が曖昧

以下の役割が明確に分離されていません。

- 作成者
- 更新担当者
- レビュー担当者

### 更新漏れが発生する

以下のような情報更新に対するフォローが属人的になっています。

- 製品アップデート
- 新機能追加
- EOL／EOS情報更新

## 目的

DLMは、以下の実現を目的とします。

- ドキュメントの最新性維持
- 更新漏れの防止
- 更新状況の可視化
- 更新履歴の記録
- レビュー品質の向上
- 属人化の解消

## 主要機能

### 1. ドキュメント台帳

管理対象となる資料を一元管理します。

| 管理項目 | 内容 |
| --- | --- |
| ドキュメント名 | 資料の名称 |
| モジュール | 関連する製品・機能 |
| カテゴリ | ドキュメント種別 |
| Boxリンク | 資料本体へのリンク |
| サポートサイトリンク | 関連情報へのリンク |
| 管理ロール | 管理を担当する組織・役割 |
| 管理者 | 資料の管理責任者 |
| 更新目的 | 更新の目的や背景 |
| メモ | 補足情報 |

### 2. 更新依頼管理

更新が必要なドキュメントに対して、更新タスクを発行します。

登録項目：

- 更新理由
- 優先度
- 更新期限
- 担当者
- レビュー担当者

### 3. ステータス管理

ドキュメントの更新状況を可視化します。

| ステータス | 説明 |
| --- | --- |
| 未着手 | 更新依頼のみ作成された状態 |
| 対応中 | 更新作業中 |
| レビュー待ち | レビューを依頼している状態 |
| 差し戻し | 指摘を受けて修正している状態 |
| 完了 | 更新内容を正式版へ反映済み |
| 保留 | 対応を見送っている状態 |

### 4. 更新履歴管理

すべての更新履歴を保存し、変更内容を追跡可能にします。

記録内容：

- 更新日時
- 更新者
- 変更内容
- レビュー者
- 承認者
- 完了日時

### 5. コメント機能

ドキュメントや更新依頼ごとに、関係者間の議論を記録します。

コメント例：

- 修正提案
- レビュー指摘
- 補足情報
- 更新理由

### 6. ダッシュボード

ドキュメントの管理状況を集約して可視化します。

表示項目：

- 総ドキュメント数
- 更新待ち件数
- レビュー待ち件数
- 期限超過件数
- 担当者別件数
- モジュール別件数

### 7. 通知機能

Microsoft Teamsとの連携を想定し、更新作業に関する通知を送信します。

通知例：

- 更新依頼の作成
- 担当者のアサイン
- レビュー依頼
- 期限超過
- 更新完了

## 権限モデル

| ロール | 主な管理対象 |
| --- | --- |
| SE | 製品紹介資料、提案資料、POV資料 |
| PE | 操作手順書、技術資料 |
| CSM／PSM | 導入手順書、導入ガイド |

各ロールが対象ドキュメントの管理責任を持ち、更新担当者やレビュー担当者と連携してライフサイクルを管理します。

## 想定ワークフロー

```text
更新依頼
  ↓
担当者アサイン
  ↓
更新作業
  ↓
レビュー依頼
  ↓
承認・正式版反映
  ↓
更新履歴を保存
```

## フロントエンドMVPの起動方法

### 必要環境

- Node.js 18以上

### 起動

```bash
npm start
```

起動後、ブラウザで以下へアクセスしてください。

```text
http://localhost:3000
```

現在はバックエンドを持たないモック版です。画面上で作成した更新依頼は、ブラウザを再読み込みするとリセットされます。

## Dockerでの起動

DockerとDocker Compose v2が利用できる環境では、以下で起動できます。

```bash
docker compose up -d --build
```

起動後、ブラウザで `http://localhost:3000` を開いてください。

公開ポートを変更する場合：

```bash
DLM_PORT=8080 docker compose up -d --build
```

## CI/CD

`.github/workflows/deploy.yml` に、セルフホストランナーを利用した自動デプロイを定義しています。

`main` ブランチへ変更がプッシュされると、以下を自動実行します。

1. Docker環境の確認
2. コミットSHAをタグにしたアプリケーションイメージのビルド
3. Docker Composeによるコンテナの差し替え
4. コンテナのヘルスチェック

### セルフホストランナーの要件

- GitHub Actionsのラベル：`self-hosted`
- Docker Engine
- Docker Compose v2
- Dockerを実行できる権限
- アプリ公開用ポート（既定値：`3000`）

ポートを変更する場合は、GitHubリポジトリの **Settings → Secrets and variables → Actions → Variables** に `DLM_PORT` を登録してください。

### デプロイ先へのアクセス

既定のアクセスURLは以下です。

```text
http://<セルフホストランナーのIPアドレス>:3000
```

Linuxランナーでは、次のコマンドでIPアドレスを確認できます。

```bash
hostname -I
```

別PCからアクセスする場合は、ランナーのOSやクラウド環境のファイアウォールでTCPポート `3000` を許可してください。
